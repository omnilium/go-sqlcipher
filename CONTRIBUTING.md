# Contributing to go-sqlcipher

Thanks for your interest. This guide covers how to work in this repo; the [Code of Conduct](./CODE_OF_CONDUCT.md) applies to all participation.

## Ground rules

Be respectful and assume good faith. Open an issue before starting any non-trivial change so we can agree the approach — this repo is a maintained fork with an unusual structure (see below), and not every change belongs here. Keep pull requests focused on one logical change.

## What belongs here

`github.com/omnilium/go-sqlcipher` is Omnilium's fork of [`mattn/go-sqlite3`](https://github.com/mattn/go-sqlite3), carried to provide a **SQLCipher** at-rest-encrypted SQLite driver. Almost the entire tree is vendored: the Go binding layer tracks upstream mattn **release tags**, and the C amalgamation (`sqlite3-binding.c/.h`, `sqlite3ext.h`) is the **SQLCipher** amalgamation, regenerated by the `upgrade/` tool — neither is hand-edited here. Omnilium's own contribution is a deliberately small patch stack: the SQLCipher codec/crypto cgo wiring, the `_key` + cipher-tuning DSN mechanism, the module-path and Go-floor foundational patches, and the carried lint/refactor fixes that hold the whole tree to our quality bar.

So before you build: a generic SQLite binding fix or a new SQLite feature almost certainly belongs **upstream at `mattn/go-sqlite3`**, where it benefits everyone and reaches us on the next resync. What belongs *here* is the SQLCipher integration, the OpenSSL provider wiring, the supported-target story (Linux x86-64/ARM64 and Apple Silicon macOS), and keeping the patch stack rebasing cleanly. When in doubt, open an issue first.

> **Heads-up on history:** `master` is force-pushed on every upstream resync (the rebase rewrites SHAs) — this is expected and correct for a vendored mirror. Don't pin a branch pseudo-version; pin a fork tag. See [CLAUDE.md](./CLAUDE.md) for the full maintenance model.

## Getting set up

This is a **CGo** build. You need a C toolchain and **OpenSSL `libcrypto`** (dev headers + library) installed — the SQLCipher codec links `-lcrypto` and there is no self-contained build.

```sh
git clone https://github.com/omnilium/go-sqlcipher.git
cd go-sqlcipher
go build ./...
```

[CLAUDE.md](./CLAUDE.md) is the authority for the full command set, the supported platforms, and the upstream-resync procedure.

## How the codebase is organized

- **Go binding layer** — `sqlite3.go` and the `sqlite3_*.go` files (the `database/sql` driver, plus the `sqlite_*` build-tag opt-ins). Inherited from mattn; held to our lint bar.
- **SQLCipher integration** — `sqlite3_crypto_openssl.go`, `sqlite3_func_crypt.go`, and the `_key` / cipher-tuning DSN handling inside `sqlite3.go`. This is the heart of the fork.
- **C amalgamation** — `sqlite3-binding.c/.h`, `sqlite3ext.h`: generated SQLCipher, **not** edited by hand.
- **`upgrade/`** — the tool that regenerates the amalgamation from a pinned SQLCipher tag.

## Testing

Get the core gates green before opening a PR — these are what CI enforces:

```sh
go build ./...
golangci-lint run ./...      # golangci-lint v2, config in .golangci.yaml
go test -race ./...
govulncheck ./...
```

The whole tree — vendored upstream code included — is held to the lint bar; findings are fixed, not exempted. When a change touches tag-gated code, a parser/decoder boundary, or the fork's key-quoting, also run the **feature-tag** suite (`go test -tags <tag> ./...` for the relevant tag) and the **fuzz smoke** (`go test -run '^$' -fuzz '^FuzzName$' -fuzztime=30s .`). Follow [test-driven development](https://github.com/omnilium): write the failing test first.

## Opening a pull request

One logical change per PR, with a clear description of what and why. The build, lint, race tests, and vuln scan must be green. If your change alters anything `README.md` or `CLAUDE.md` documents — the upstream base, the build commands, the SQLCipher integration, the versioning scheme — update those files in the same PR.

### Commits

Follow [Conventional Commits 1.0.0](https://www.conventionalcommits.org/), per the Omnilium house style. Keep the patch stack small, self-contained, and clearly labeled so it rebases cleanly onto each upstream release tag.

## Releases

The fork has its **own independent semver line starting at `v0.1.0`**, not aligned to upstream's `v1.14.x` numbers. Releases are cut by the maintainers as fork **tags** after a resync lands and the gates are green; the changelog lives in [CHANGELOG.md](./CHANGELOG.md). Consumers pin a fork tag (never a branch pseudo-version — a resync rewrites `master`'s history).
